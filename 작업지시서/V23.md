# 작업지시서 V23 (보고서)

**작성일:** 2026-04-03
**목적:** 전개도 컷팅 오류 및 시뮬레이션 비율 찌그러짐 현상 원인 파악 및 해결 방안

---

## 1. 문제 현상 분석

제공해주신 스크린샷에서 확인된 사항은 다음과 같습니다.
- **현상 1 (전개도 손상):** 아트보드에서 각 뷰가 밀리거나 일부 잘리며 불규칙한 여백이 발생합니다.
- **현상 2 (형태 왜곡):** 시뮬레이션 생성 결과물이 실제 건물 비율과 다르게 극단적으로 뚱뚱해집니다.

**[근본 원인 파악: 단순 4등분 크롭 로직의 모순]**
* AI는 `V21` 및 `V18` 프로토콜 규칙을 충실히 수행하여, **건물의 실제 비례**(`[Depth, Width, Depth, Width]`)대로 각기 다른 픽셀 넓이를 가진 T-Shape 시트를 정확하게 생성해냈습니다.
* 그러나, 정작 이 시트를 분해하는 프론트엔드의 `cropElevationFromSheet` 함수는 건물 비율을 완전히 무시한 채, **단순 무식하게 전체 폭을 1/4 (25%)씩 등분**하여 프레임으로 잘라내고 있었습니다.
* 이로 인해 W와 D가 다른 직사각형 건물의 경우, 잘린 이미지 안에 앞면과 옆면이 반씩 섞여버리게 됩니다. AI는 이 "망가진 기형 도면"을 `IMAGE 2, 3`으로 넘겨받아 렌더링의 기하학 뼈대로 사용했기 때문에, 최종 결과물이 뚱뚱하고 그로테스크하게 찌그러졌던 것입니다.

---

## 2. 해결 방안 (수행 계획)

건축 비례의 무결성을 복원하기 위해 크롭 함수를 "비례 기반의 스케일 컷(Scale-Cut)"으로 리팩토링합니다.

1. **상대 비례 연산 주입:** `cropElevationFromSheet` 함수의 인자에 `bldgRatio {width, depth, height}` 정보를 추가합니다.
2. **동적 좌표 추출 연산:** 전체 넓이가 4등분이 아닌, `(W*2 + D*2)`의 합계로 환산된 단위 Unit 값을 도출합니다.
   - Column 0: Width = Unit * Depth
   - Column 1: Width = Unit * Width
   - 이 방식으로 각 면의 실제 시작 좌표(startX, startY)와 Width, Height를 픽셀 단위로 완벽하게 계산하여 크롭합니다.
3. **호출부 업데이트:** `App.tsx` 내에서 렌더링 시트를 자르는 두 군데 실행부(`generateSitePlan`, `generateElevationFromImage` 과정)에 이 `bldgRatio`를 파라미터로 넘기도록 조치합니다.

이 패치가 적용되면 각 면이 한 치의 오차 없이 깔끔하게 분리되어 아트보드 정렬 오류와 최종 시뮬레이션 왜곡 현상을 동시에 박멸할 수 있습니다.
